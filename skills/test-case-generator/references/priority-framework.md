# 优先级框架 — 决策树与验证规则

> 为每个测试用例分配 P0 / P1 / P2 优先级，目标比例：P0 约 15-20%，P1 约 40-50%，P2 约 30-40%。

---

## 1. 优先级定义

| 优先级 | 标准 | 测试套件 |
|--------|------|---------|
| **P0** | 核心业务路径中断 = 产品不可用 | smoke + regression + full |
| **P1** | 重要功能异常 = 影响核心体验 | regression + full |
| **P2** | 边缘场景 = 影响少数用户或非关键体验 | full |

---

## 2. 优先级决策树

```
Q1: 该用例失败是否意味着核心业务路径（登录/支付/核心CRUD）完全不可用？
  ├─ YES → P0
  └─ NO
      Q2: 该用例覆盖的功能是否在 80% 用户的日常使用路径上？
        ├─ YES → P1
        └─ NO
            Q3: 该用例是否覆盖有安全、数据完整性或法规合规影响的场景？
              ├─ YES → P1
              └─ NO → P2
```

### 补充规则（覆盖决策树）

| 场景 | 优先级 |
|------|--------|
| 系统启动/初始化的 Happy Path | P0 |
| 认证/授权（登录、权限控制）的正常路径 | P0 |
| 数据丢失或损坏场景 | P0 |
| 连续失败导致账号锁定 | P1 |
| 关键错误提示（用户能看到的友好错误） | P1 |
| 常见的边界值（密码长度、文件大小） | P1 |
| 特殊字符/XSS 注入防御 | P1 |
| 非关键 UX 细节（动画、颜色、tooltip） | P2 |
| 极端边界值（超过实际使用的上限） | P2 |
| 兼容性、多语言显示（非英文字符） | P2 |
| 并发操作（偶发场景） | P2 |

---

## 3. 比例验证规则

生成完成后，统计 `## Merged Test Case List` 中的 P0/P1/P2 数量：

```
total = P0_count + P1_count + P2_count

验证：
  P0_ratio = P0_count / total
  P1_ratio = P1_count / total
  P2_ratio = P2_count / total

目标范围：
  0.15 ≤ P0_ratio ≤ 0.20  （15-20%）
  0.40 ≤ P1_ratio ≤ 0.50  （40-50%）
  0.30 ≤ P2_ratio ≤ 0.40  （30-40%）

如果比例偏差 > 5%：
  P0 过多 → 检查是否有非核心路径被标记为 P0
  P1 不足 → 检查错误提示和边界值用例是否被标记为 P2
  P2 不足 → 检查边缘场景覆盖是否充分
```

---

## 4. 常见误判场景

| 错误分类 | 正确分类 | 说明 |
|---------|---------|------|
| 把所有 Happy Path 都标 P0 | P0 只给核心路径 | 非关键功能的 Happy Path 应为 P1 |
| 把错误提示用例标 P2 | 应为 P1 | 用户看到的错误提示是产品质量的一部分 |
| 把权限检查标 P2 | 应为 P1 | 权限错误影响安全，至少 P1 |
| 把重复提交检查标 P2 | 应为 P1 | 数据一致性问题 |
| 把 XSS 防御标 P2 | 应为 P1 | 安全相关，强制 P1 |

## 5. 与 tags 的对应关系

| 优先级 | 自动添加的 tags |
|--------|---------------|
| P0（positive） | `@P0`, `@smoke`, `@regression`, `@full` |
| P0（negative/boundary/error） | `@P0`, `@regression`, `@full` |
| P1 | `@P1`, `@regression`, `@full` |
| P2 | `@P2`, `@full` |

> `@smoke` 只给 P0 + positive 组合：这是最精简的"冒烟测试"套件，核心正向流程快速验证。
