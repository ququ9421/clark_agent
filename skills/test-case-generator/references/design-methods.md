# 6 种测试设计方法 — 详细步骤与完整示例

> 每种方法都包含：适用场景 → 执行步骤 → Markdown 输出示例 → Gherkin BDD 示例。
> 所有 6 种方法必须逐一检查；不适用时写 `N/A — 原因：{说明}`。

---

## 方法 1：等价类划分（Equivalence Partitioning）

### 适用场景
任何有输入字段的功能：表单、搜索、筛选、参数验证。

### 执行步骤

```
1. 列出所有输入字段
2. 对每个字段划分有效类和无效类：
   - 有效类：能产生正确处理结果的输入集合
   - 无效类：应产生错误或被拒绝的输入集合
3. 为每个等价类设计一个代表值
4. 规则：一个用例可覆盖多个有效类，但只能覆盖一个无效类
5. 特殊类型：
   - 数字字段：正数类、零、负数类、非数字类
   - 字符串字段：空字符串类、有效格式类、无效格式类、超长类
   - 枚举字段：每个合法值为一个有效类，非枚举值为无效类
```

### Markdown 输出示例

```markdown
## Method 1: 等价类划分

**等价类划分表**

| 字段 | 有效类 | 无效类 |
|------|--------|--------|
| 邮箱 | 标准格式（user@domain.com） | 缺少@符号、缺少域名、空值、超过254字符 |
| 密码 | 8-20位含大小写字母和数字 | 纯数字、纯字母、少于8位、超过20位 |

**TC-PRD-LOGIN-001**: 使用有效邮箱和密码登录成功
- **优先级:** P0
- **测试类型:** 等价类划分
- **前置条件:** 用户 test@example.com 已注册，密码为 Test@12345
- **操作步骤:** 1. 在邮箱输入框输入"test@example.com" 2. 点击"继续" 3. 输入密码"Test@12345" 4. 点击"登录"
- **预期结果:** 成功登录，跳转到 /dashboard，页面显示用户名称
- **测试数据:** 邮箱: test@example.com | 密码: Test@12345

**TC-PRD-LOGIN-002**: 使用缺少@符号的邮箱登录失败
- **优先级:** P1
- **测试类型:** 等价类划分
- **前置条件:** 当前在登录页
- **操作步骤:** 1. 在邮箱输入框输入"invalidemail.com" 2. 点击"继续"
- **预期结果:** 邮箱输入框下方显示红色错误提示"请输入有效的邮箱地址"，无法进入下一步
```

## 方法 2：边界值分析（Boundary Value Analysis）

### 适用场景
有数值范围、字符长度限制、分页、数量上下限的功能。

### 执行步骤

```
1. 找出所有有范围约束的字段
2. 对每个范围，确定 4 个边界点：
   - 最小值（min）
   - 最小值 + 1（min+1）
   - 最大值 - 1（max-1）
   - 最大值（max）
   可选：低于最小值（min-1）、高于最大值（max+1）
3. 常见边界类型：
   - 字符长度：空(0)、1字符、最大-1、最大、最大+1
   - 数量：0、1、最大-1、最大、最大+1
   - 分页：第1页、第2页、最后一页、超出页数
   - 金额：0.00、0.01、999999.99、1000000.00
4. 每个边界点设计一个用例
```

### Markdown 输出示例

```markdown
## Method 2: 边界值分析

**边界值分析表**

| 字段 | 规则 | 测试值 | 期望结果 |
|------|------|--------|---------|
| 密码长度 | 8-20位 | 7位 | 拒绝，显示"密码至少需要8个字符" |
| 密码长度 | 8-20位 | 8位 | 接受 |
| 密码长度 | 8-20位 | 20位 | 接受 |
| 密码长度 | 8-20位 | 21位 | 拒绝，显示"密码不能超过20个字符" |

**TC-PRD-LOGIN-005**: 密码恰好 8 位时注册成功（最小有效边界）
- **优先级:** P1
- **测试类型:** 边界值分析
- **前置条件:** 当前在注册页
- **操作步骤:** 1. 填写邮箱"boundary@test.com" 2. 输入密码"Test@123"（恰好8位） 3. 点击"注册"
- **预期结果:** 注册成功，跳转到邮箱验证页
- **测试数据:** 密码: Test@123 (8位)

**TC-PRD-LOGIN-006**: 密码为 7 位时注册被拒绝（低于最小边界）
- **优先级:** P1
- **测试类型:** 边界值分析
- **前置条件:** 当前在注册页
- **操作步骤:** 1. 填写邮箱"boundary2@test.com" 2. 输入密码"Test@12"（7位） 3. 点击"注册"
- **预期结果:** 密码输入框下方显示红色提示"密码至少需要8个字符"，注册被阻止
- **测试数据:** 密码: Test@12 (7位)
```

## 方法 3：判定表 / 因果图（Decision Table / Cause-Effect Graph）

### 适用场景
多条件组合场景：权限控制、业务规则、状态组合、条件显示/隐藏。

### 执行步骤

```
1. 列出所有独立条件（Causes）
2. 列出所有可能的结果（Effects）
3. 枚举有意义的条件组合（可剪枝逻辑上不可能的组合）
4. 建议格式：
   - 条件 ≤ 4 个时：全排列（2^n 行，可剪枝）
   - 条件 > 4 个时：只覆盖"有影响的"组合

构建判定表：
  条件1（T/F）× 条件2（T/F）× 条件3（T/F）→ 结果

示例：已登录(T/F) × 有权限(T/F) → 可访问(T/F) + 跳转行为
```

### Markdown 输出示例

```markdown
## Method 3: 判定表 / 因果图

**判定表**

| 已登录 | 有编辑权限 | 数据状态 | 期望行为 |
|--------|-----------|---------|---------|
| ✓ | ✓ | 草稿 | 显示编辑按钮，可编辑 |
| ✓ | ✓ | 已发布 | 显示编辑按钮，但保存时提示需创建新版本 |
| ✓ | ✗ | 任意 | 隐藏编辑按钮，只读模式 |
| ✗ | — | 任意 | 跳转登录页，参数带 redirect_url |

**TC-PRD-EDIT-001**: 已登录且有权限时编辑草稿状态文档
- **优先级:** P0
- **测试类型:** 判定表
- **前置条件:** 通过 UI 创建草稿状态文档"TestDoc-{timestamp}"；以具有编辑权限的账号登录
- **操作步骤:** 1. 打开文档详情页 2. 点击"编辑"按钮 3. 修改标题为"Updated-{timestamp}" 4. 点击"保存"
- **预期结果:** 文档标题更新成功，详情页显示新标题，显示"已保存"提示
- **测试数据:** 初始标题: TestDoc-{timestamp} | 修改后: Updated-{timestamp}
```

## 方法 4：状态转换（State Transition Testing）

### 适用场景
对象有明确状态机的功能：工单状态、订单流程、文档审批、任务生命周期。

### 执行步骤

```
1. 识别对象的所有状态（State）
2. 识别所有合法的状态转换（Transition）
3. 识别所有非法转换（Illegal Transition）
4. 设计测试用例：
   - 正向路径：覆盖每条合法转换路径
   - 逆向路径：尝试非法转换，验证系统拒绝
5. 注意：循环路径（如可重新打开的工单）需要专门测试

状态图示例：
  草稿 → [提交] → 待审核 → [批准] → 已发布
                          → [拒绝] → 草稿
  已发布 → [下架] → 已归档
  已归档 → [恢复] → 草稿
```

### Markdown 输出示例

```markdown
## Method 4: 状态转换

**状态转换表**

| 当前状态 | 触发动作 | 目标状态 | 是否合法 |
|---------|---------|---------|---------|
| 草稿 | 提交审核 | 待审核 | ✓ |
| 待审核 | 批准 | 已发布 | ✓ |
| 待审核 | 拒绝 | 草稿 | ✓ |
| 已发布 | 下架 | 已归档 | ✓ |
| 已归档 | 恢复草稿 | 草稿 | ✓ |
| 已发布 | 直接提交审核 | — | ✗（非法） |
| 已归档 | 直接发布 | — | ✗（非法） |

**TC-PRD-STATE-001**: 文档从草稿成功提交审核
- **优先级:** P0
- **测试类型:** 状态转换
- **前置条件:** 通过 UI 创建草稿状态文档"Draft-{timestamp}"
- **操作步骤:** 1. 打开文档详情页 2. 点击"提交审核"按钮 3. 在确认对话框点击"确认"
- **预期结果:** 文档状态标签变为"待审核"，"提交审核"按钮消失，页面显示"已提交，等待审核"提示
- **测试数据:** 文档标题: Draft-{timestamp}
```

## 方法 5：场景法（Scenario Method）

### 适用场景
需要覆盖完整用户旅程的端到端流程：注册→登录→使用→退出；新建→编辑→删除；上传→处理→下载。

### 执行步骤

```
1. 识别主要用户旅程（Happy Path）
2. 识别替代路径（Alternative Path）
3. 识别异常路径（Unhappy Path）
4. 每个场景必须包含：
   - 完整前置条件（可执行）
   - 完整操作序列（多步骤）
   - 中间状态验证（关键节点）
   - 最终状态验证（业务结果）
5. 场景数量建议：1 个 Happy Path + 2-3 个异常路径
```

### Markdown 输出示例

```markdown
## Method 5: 场景法

**TC-PRD-FLOW-001**: 完整任务创建→编辑→完成流程（Happy Path）
- **优先级:** P0
- **测试类型:** 场景法
- **前置条件:** 用户已登录，当前在任务列表页
- **操作步骤:**
  1. 点击"新建任务"按钮，填写任务名称"Test-Flow-{timestamp}"，点击"创建"
  2. 确认新任务出现在列表顶部，状态为"待处理"
  3. 点击该任务进入详情页，点击"编辑"，修改描述为"已更新-{timestamp}"，保存
  4. 返回列表页，点击"完成"按钮将任务标记为完成
  5. 确认任务状态变为"已完成"
- **预期结果:**
  - 步骤2：列表包含新任务，状态为"待处理"，显示创建时间
  - 步骤3：详情页描述更新，显示"已保存"提示
  - 步骤5：任务状态标签变为"已完成"（绿色），完成时间已记录
- **测试数据:** 任务名: Test-Flow-{timestamp} | 描述: 已更新-{timestamp}
```

## 方法 6：错误猜测（Error Guessing）

### 适用场景
所有功能，特别是历史上容易出错的场景：空值提交、特殊字符、并发、重复操作。

### 执行步骤

```
优先检查以下高频错误场景：

1. 空值/边界输入：
   - 提交全空表单
   - 只填必填字段中的一个
   - 输入只有空格的内容

2. 特殊字符：
   - SQL 注入：' OR 1=1 --
   - XSS：<script>alert(1)</script>
   - 特殊符号：!@#$%^&*()、中日韩字符、Emoji

3. 操作时序：
   - 连续双击提交按钮（重复提交）
   - 表单提交中途刷新页面
   - 网络慢时提交（可配合 DevTools 节流）

4. 权限边界：
   - 直接访问无权限的 URL
   - 修改请求中的用户 ID
   - 退出登录后使用浏览器回退按钮

5. 并发操作：
   - 两个用户同时编辑同一条数据
   - 快速连续删除同一条数据

6. 数据一致性：
   - 删除后刷新列表是否立即消失
   - 编辑后其他页面的显示是否同步更新
```

### Markdown 输出示例

```markdown
## Method 6: 错误猜测

**TC-PRD-ERR-001**: 连续双击"创建"按钮不产生重复数据
- **优先级:** P1
- **测试类型:** 错误猜测
- **前置条件:** 用户已登录，当前在任务创建页
- **操作步骤:** 1. 填写任务名称"DoubleClick-{timestamp}" 2. 快速连续双击"创建"按钮
- **预期结果:** 任务列表中只出现一条名为"DoubleClick-{timestamp}"的任务，不产生重复数据；第二次点击被忽略或提示"正在提交"
- **测试数据:** 任务名: DoubleClick-{timestamp}

**TC-PRD-ERR-002**: 输入 XSS 内容不被执行
- **优先级:** P1
- **测试类型:** 错误猜测
- **前置条件:** 用户已登录，当前在任务创建页
- **操作步骤:** 1. 在任务名称输入框输入 `<img src=x onerror=alert('XSS')>` 2. 点击"创建" 3. 打开任务详情页
- **预期结果:** 任务名称显示为纯文本 `<img src=x onerror=alert('XSS')>`，页面不弹出 alert 对话框，XSS 未执行
- **测试数据:** 任务名: `<img src=x onerror=alert('XSS')>`
```
